# Action-Ready Synthesis: c11-104-plan

> **Reviewer-coverage note:** Gemini was rate-limited during this review pack. All consensus claims below are taken across the **6 successful reviews (2 models × 3 lenses: Claude+Codex Standard, Adversarial, Evolutionary)**, not the full 9-review slate. "Multi-reviewer consensus" therefore means 2+ of those 6 reviewers, not 2+ of 9. Treat the consensus threshold accordingly.

## Verdict

**revise-then-proceed**

Reviewer verdicts ranged from "ready with one round of stitch-to-existing-code clarification" (Standard-Claude) to "Needs revision before execution" (Standard-Codex) and "I am concerned" (Adversarial-Codex). The shared signal across all six reviews is the same: the design idea is right, but the plan does not reconcile with three existing systems in the c11 codebase (the `TabManager` git probe, the `panelGitBranches`/`panelDirectories` sidebar branch row, and the existing `sidebarShowBranchDirectory` setting), under-specifies several correctness-load-bearing details (cache key for linked worktrees/submodules, stale-result guards, derived-source persistence/socket policy, error/timeout behavior), and asserts a few file paths that do not exist in the repo. Biasing to the more cautious verdict per the rules of thumb.

The architecture survives this pass intact. None of the validated findings demand a re-architecture; they demand an integration-map clarification, a few correctness fixes in the resolver and cache, a settled policy on `derived`-source persistence and writability, and path corrections in the BUILDPLAN. After those land, this plan is executable.

## Apply by default

### Blockers (plan is not yet executable as written)

- **B1: No integration map for existing `TabManager` git probe / `panelGitBranches` / `panelDirectories` / `sidebarShowBranchDirectory` toggle**
  - Where in the plan: Whole document — most acutely in "Implementation notes → Data layer," "BUILDPLAN → Files (initial map)," "Settings" section, and AC14.
  - Problem: The plan reads as if c11 has no existing git resolution, no per-panel branch/directory rendering, and no Sidebar Settings toggle for branch/directory display. All three exist (`TabManager.initialWorkspaceGitMetadataSnapshot` + `scheduleInitialWorkspaceGitMetadataRefresh` + `applyWorkspaceGitMetadataSnapshot`; `panelDirectories` + `panelGitBranches` + `SidebarBranchOrdering` rendered in `TabItemView`; `@AppStorage("sidebarShowBranchDirectory")` Settings toggle). Implemented literally, the delegator will produce a parallel git pipeline that races the existing one, a sidebar that shows branch info twice, and a Settings panel with two near-identical default-on "Show branch" toggles.
  - Revision: Add an "Integration map" subsection to the BUILDPLAN (Part 2) that explicitly answers (a) whether the new resolver subsumes / extends / runs alongside the existing `TabManager` probe — recommendation across reviewers is to extend the existing probe so there's one queue, one cache, one generation-token system, one source of truth for branch/dirty/PR; (b) whether the new chips replace, augment, or coexist with the existing "Branch + Directory" sidebar row, and what the resulting behavior of `sidebarShowBranchDirectory` is — recommendation is to commit to one toggle name/behavior, not two; (c) whether the existing dirty-marker and PR pill that the legacy row renders are preserved on the new chips, retired, or kept on the legacy row that still ships alongside. Pin the decision in writing before the delegator codes.
  - Sources: Standard-Claude (executive summary + weaknesses #1, #7, #8 + readiness verdict + questions 1–5); Standard-Codex (weaknesses #2 + alternative A/B + question 6, 7, 16); Adversarial-Codex (blind spots paragraphs 1–2 + question 4, 6 + hindsight); Evolutionary-Claude (concrete suggestion S2 implies coordinator-not-duplicate).

- **B2: `(cwd, mtime(.git/HEAD))` cache key is wrong for linked worktrees and submodules — the exact cases this ticket exists to support**
  - Where in the plan: "Implementation notes → Hot-path discipline" bullet "Cache per cwd. `(cwd, mtime-of-.git/HEAD)` is the cache key"; BUILDPLAN "Cache key `(cwd, mtime-of-.git/HEAD)`"; AC12.
  - Problem: In a linked worktree, `<worktree>/.git` is a *file* pointing at `<common-dir>/worktrees/<name>/`, and the real HEAD that changes on branch checkout is `<common-dir>/worktrees/<name>/HEAD`. In a submodule, `<sub>/.git` is similarly a gitfile. `stat`-ing `<cwd>/.git/HEAD` in those cases is either ENOENT or watches the wrong file. AC12's "touch `.git/HEAD`" test will pass against a main-checkout fixture and silently fail to exercise the cases the feature is supposed to serve.
  - Revision: Replace the cache key with one built from git-resolved paths. The recommended algorithm across reviewers: invoke `git rev-parse --git-path HEAD` (and/or `--git-dir` + `--git-common-dir`) once during resolution, use the resolved HEAD path's mtime (or `DispatchSource`/FSEvents subscription against it) as the invalidation signal, key the cache against the resolved gitdir/common-dir rather than `<cwd>/.git/HEAD`. Update AC12 to require the test to exercise at least one linked-worktree fixture and one submodule fixture, and to assert the cache invalidates when the *resolved* HEAD file changes — not just `<cwd>/.git/HEAD`.
  - Sources: Standard-Codex (weakness #4, "Suggested Plan Patch"); Adversarial-Claude (assumption A2 + blind spot #11 "what about external git checkout" + assumption "`.git/HEAD` exists on every code path" + hard question 1, 4); Adversarial-Codex (load-bearing assumption #2 + challenged decisions "cache key should not be mtime-centered" + hard questions 9, 10).

- **B3: Submodule resolution sequence is incorrect — `--show-toplevel` from a submodule returns the submodule root, not the superproject**
  - Where in the plan: BUILDPLAN "Resolution sequence" steps 1–5; refined-spec decision 7 ("Submodule context: stacked two rows"); AC6.
  - Problem: Step 1 (`--show-superproject-working-tree`) correctly returns the superproject path *only* when invoked from inside a submodule. Step 2 (`--show-toplevel`) from the same cwd returns the *submodule's* toplevel, not the superproject's. So the sequence as written cannot actually populate the "row 1: superproject worktree+branch" the refined decision requires. The two contexts need separate passes — one resolution run with cwd inside the submodule (for the inner row) and a second resolution run with cwd at the superproject working tree (for the outer row). Submodule name source is also unspecified (basename vs `.gitmodules` configured name vs path-from-superproject-root).
  - Revision: In the BUILDPLAN, replace the single-pass resolution sequence with a two-pass algorithm: (1) detect submodule via `--show-superproject-working-tree`; if non-empty, treat that path as the outer worktree and run a full resolution against it (worktree path, kind: main vs linked, branch via `symbolic-ref` or detached SHA, using git-resolved gitdir); (2) run a separate full resolution against the original submodule cwd for the inner row. Pin the submodule display name source explicitly (recommendation: configured submodule name from the superproject's `.gitmodules`, falling back to path-from-superproject-root, falling back to basename). Update AC6 to assert correctness of *both* rows' worktree path + branch independently, and add a separate AC for "linked worktree containing a submodule with a detached inner HEAD" which is the realistic combined case (ghostty inside c11 is the operator's canonical example).
  - Sources: Standard-Codex (architectural assessment "Submodule handling is under-specified and partly wrong" + weakness #5 + questions 11–12); Adversarial-Codex (blind spots paragraph "Submodule handling is under-specified and partly wrong" + reality stress test disruption 3 + hard questions 12–13); Adversarial-Claude (blind spot #7 "Submodule edge cases" + hindsight "we accepted tests that touched the happy git topology but missed real worktree/submodule gitfile behavior").

- **B4: Stale-async-result handling unspecified — no generation token / expected-cwd guard before applying resolver output**
  - Where in the plan: BUILDPLAN "OSC 7 cwd-update handler triggers resolution off-main… Update the rendered metadata when results arrive"; AC10.
  - Problem: When cwd changes A → B → C in rapid succession (shell scripts, `cd a && cd b && cd c`, agent `pushd`/`popd`), the resolver fires multiple times. Without a per-surface generation token and an expected-cwd check before applying each result, a late-returning resolver for cwd A can clobber the freshly-rendered chips for cwd C. The existing `TabManager` probe already uses this pattern (generation tokens + expected-directory check). The plan does not require the new resolver to do the same.
  - Revision: Add an explicit BUILDPLAN clause: "Each resolver invocation is tagged with a per-surface generation counter and the cwd it was kicked for. Before applying any result, verify the surface still exists, the generation counter matches the surface's current generation, and the surface's current cwd still equals the resolver's input cwd. If any check fails, drop the result. Reuse the existing pattern from `TabManager.scheduleInitialWorkspaceGitMetadataRefresh`." Add an AC: "resolver kicked for cwd A and a subsequent kick for cwd B both fire; if A returns after B's result has been applied, A's result is dropped (not rendered)." This is testable as a logic test with a fake clock + injected resolver.
  - Sources: Standard-Codex (weakness #3 + "Suggested Plan Patch" wording); Adversarial-Codex (blind spots "The plan does not specify stale-result cancellation" + reality stress test #2 + hard question 7); Adversarial-Claude (blind spot #2 "Concurrent OSC 7 storms" + hard question 16).

- **B5: `derived`-source policy is undefined across socket / persistence / clear semantics**
  - Where in the plan: SPEC "Precedence: introduce a new source tag, `derived`, ranked below `osc`"; "Derive, don't store. … Do not write them to the surface manifest as `explicit` entries"; BUILDPLAN "Introduce a new source tag `derived`"; AC13.
  - Problem: The plan introduces `derived` as a new `MetadataSource` tier without deciding three load-bearing policy questions. (a) **Socket writability:** can a CLI/socket client write `source: derived` via `surface.set_metadata`, or is `derived` accepted only from c11-internal writers? If not gated, agents can write whatever they like and call it "derived," nullifying the meaning. (b) **Persistence:** `SurfaceMetadataStore` is captured into session/workspace snapshots and restored on resume. The plan says "store nothing, recompute on cwd change" — but if derived keys are written through the same store, they will persist by default unless capture and restore paths explicitly exclude `source == .derived`. (c) **Clear semantics:** existing c11 clear rules allow a lower-tier writer to clear only if the current value is at-or-below the caller's tier. When cwd leaves a git repo, *who* issues the `clear_metadata` for `worktree`/`branch`, with what source? If derived can't clear derived (or anything above it), chips will stick after `cd ~/Desktop`.
  - Revision: Add a short "`derived` source policy" subsection to the BUILDPLAN that decides each of (a), (b), (c) explicitly. Reviewer-aligned recommendation: `derived` is **internal-only** (socket/CLI clients writing `source=derived` are rejected with `invalid_source`); derived keys are **excluded from snapshot capture** (and any restore code that encounters legacy persisted derived values discards them); derived writers may **clear their own derived values** (the resolver explicitly issues a clear when it determines "not in a repo"). Add ACs covering: (i) socket write with `source=derived` is rejected, (ii) snapshot capture excludes derived keys, (iii) leaving a repo clears the chips (not just hides them via stale cache). Document these in `references/metadata.md` per AC17.
  - Sources: Standard-Codex (weakness #1, #6 + alternatives B/C + questions 1–4 + "Suggested Plan Patch"); Adversarial-Codex (load-bearing assumption "derived canonical metadata can be in same write path…" + challenged decisions "derived source tier is questionable" + hard questions 1–3, 14); Adversarial-Claude (assumption A4 + "Precedence-tier additions ripple to clear semantics" #5 + hard question 6); Evolutionary-Codex (concrete suggestion #4 "Treat `derived` as c11-owned. If the socket accepts it, document whether external writers are allowed").

- **B6: Wrong test file path and wrong sidebar source directory in the BUILDPLAN**
  - Where in the plan: BUILDPLAN "Files (initial map)" — `Sources/Sidebar/`, `Tests/c11LogicTests/`, `Sources/Settings/`. AC1–AC9 verification column ("`Tests/c11LogicTests/<NewFile>.swift`"). AC15 pass condition ("File location: `Tests/c11LogicTests/...`. NOT `Tests/c11Tests/...`").
  - Problem: The repo does not have a `Tests/c11LogicTests/` directory or a `Sources/Sidebar/` directory. Swift tests live in `c11Tests/` with `c11LogicTests` distinguished by target membership in the Xcode project (per the `Testing policy` block in `CLAUDE.md`). Existing chip rendering lives in `Sources/AgentChip.swift`, `Sources/AgentChipBadge.swift`, and `Sources/ContentView.swift` (workspace row).   - Revision: In the BUILDPLAN "Files" section and across AC1–AC9, AC15, replace `Tests/c11LogicTests/` with `c11Tests/<NewFile>.swift` with `c11LogicTests` target membership (the existing convention). Replace `Sources/Sidebar/` with a note that chip-render code extends `Sources/AgentChip.swift` / `Sources/AgentChipBadge.swift` / the workspace row in `Sources/ContentView.swift` in place (delegator to confirm exact entry point during plan phase). Leave `Sources/Settings/` as a placeholder if Settings code in fact lives elsewhere — the delegator confirms during plan phase. AC15's pass condition should be "test file is a member of the `c11LogicTests` target," not "file is under `Tests/c11LogicTests/`."

- **I2: AC10's "no `git` Process invocation on main" is grep-shaped, not runtime-shaped; misses synchronous-wait pattern**
  - Where in the plan: AC10 ("Inspect the new resolver entry point: any code that calls `git ...` lives inside a `DispatchQueue.global` / `Task.detached` / equivalent off-main scope. … Reviewer confirms no `git` Process invocation on main").
  - Problem: A handler can be on-main, dispatch the git work to a global queue, and then call `.wait()` or `await` synchronously on the result — blocking main even though `Process()` itself is on a background queue. AC10 as written passes such a pattern vacuously. The plan needs a runtime-shaped assertion that the OSC 7 handler returns synchronously without awaiting the resolver, or that resolver work cannot complete on the main thread.
  - Revision: Reword AC10's pass condition: "(a) the OSC 7 handler invokes the resolver via fire-and-forget enqueue and returns synchronously without blocking on `git` output; (b) the resolver function preconditions on `dispatchPrecondition(condition: .notOnQueue(.main))` at entry, which gives a runtime crash if accidentally invoked on main; (c) a logic test invokes the resolver from main and asserts the precondition trips." Drop the grep-style "no `git` Process invocation on main" framing — keep that as a code-review aid but not as the test.
  - Sources: Standard-Claude (weakness #6 + question 13); Adversarial-Claude ("The Uncomfortable Truths #1" + hard question 12); Adversarial-Codex ("The plan's acceptance criteria can be made green without proving the feature is trustworthy" + hard question 17).

- **I3: No git-subprocess timeout / hang policy specified**
  - Where in the plan: BUILDPLAN "Resolution sequence" runs 4+ `git rev-parse`/`symbolic-ref` invocations per OSC 7 fire (8+ in the submodule case); no timeout, no fallback.
  - Problem: `git` is not pure CPU; it touches filesystem locks, index files, and can stall multi-seconds on slow disks, network mounts, repos with lock contention, or corrupted state. A hung `git` invocation will hold a background-queue slot indefinitely and stack up behind subsequent OSC 7 events. The plan never sets a per-invocation timeout, a per-surface concurrency cap, or fallback behavior when timeout fires.
  - Revision: Add to BUILDPLAN: "Each `git` invocation in the resolver runs with a wall-clock timeout (recommended: 2 seconds per invocation). On timeout, the in-flight subprocess is terminated, the resolver returns nil/empty for that surface, and the chip clears (or renders the last-known value with a configurable staleness indicator — delegator picks one). At most one resolver in-flight per surface at any time; a new OSC 7 while one is in-flight either coalesces (preferred) or supersedes-and-cancels the previous." Add an AC: "a resolver invocation whose `git` subprocess does not complete within the timeout is terminated and the resolver returns a defined timeout result (no crash, no hung queue)."
  - Sources: Adversarial-Claude (blind spot #1 "Performance budget" + #4 "git segfaults or hangs" + hard questions 2–3 + reality stress test disruption 1–2); Adversarial-Codex (load-bearing assumption #3 + challenged decisions "casual about git command latency" + reality stress test disruption 1 + hard question 8); Standard-Codex (weaknesses #3 + Suggested Plan Patch).

- **I4: Workspace-multi-surface ambiguity — which surface(s) drive sidebar chips?**
  - Where in the plan: SPEC "render-in-sidebar"; BUILDPLAN "Sidebar render."
  - Problem: A c11 workspace contains multiple panels/surfaces and the sidebar shows one row per workspace tab. The existing branch/directory row aggregates per-panel state across ordered panel IDs. The plan never decides whether the new chips reflect (a) the focused surface only, (b) all visible surfaces in the workspace, (c) an aggregate "if all surfaces share a worktree, show one chip; if they diverge, show an aggregate hint," or (d) something else. Different choices produce materially different UX. This is the kind of ambiguity that breeds rework if not pinned before code.
  - Revision: Add a BUILDPLAN clause: "In a workspace with multiple terminal surfaces, the sidebar chip row reflects [chosen policy — e.g., 'the focused surface's worktree/branch; if not yet derived, show last-known from any surface in the workspace']. Surfaces with no derived context (markdown surfaces, browser surfaces, surfaces with cwd outside a repo) contribute nothing." Pick a single policy and write it down. An AC asserting the chosen behavior in a multi-surface fixture follows naturally.
  - Sources: Standard-Codex ("The plan's Intent vs Its Execution" + weakness "Load-bearing assumption: sidebar chips are per-surface but…" + question 5); Adversarial-Codex (load-bearing assumption #5 + blind spots "render aggregation rules" + hard question 5).

- **I5: Restore-window gap unspecified — what renders between session restore and first OSC 7?**
  - Where in the plan: SPEC "Out of scope: Persistent storage of `worktree`/`branch` — restored panes will re-derive on session resume."
  - Problem: c11 snapshots include `currentDirectory` and `gitBranch` per panel. On restore, the new chips re-derive — but re-derivation is async and waits on the resolver. On a workspace with 30 surfaces, the operator will watch chips populate one-by-one over the first 1–2 seconds after restore. Worse, *some* derived data lives in the workspace snapshot (existing `gitBranch`), so the existing branch row will show a value immediately while the new chips lag — a confusing split-render. The plan doesn't address whether (a) restore should pre-warm by walking all surfaces' last-known cwd and kicking resolvers in parallel before the sidebar paints, (b) the sidebar should render last-known optimistically and reconcile, or (c) the lag is acceptable.
  - Revision: Add to BUILDPLAN: "On workspace restore, walk all restored surfaces' last-known cwd and kick the resolver for each on the background queue *before* the first sidebar paint, with a short timeout (e.g., 250ms) after which the sidebar paints whatever has resolved and lets the rest fill in async." Alternatively (simpler), require that on restore the sidebar renders the previous-session `gitBranch` (from the existing snapshot) optimistically as the new branch chip's initial value, then the resolver reconciles. Pick one and write it down.
  - Sources: Adversarial-Claude (blind spot #4 "Two-source-of-truth problems on session restore" + assumption A5 + hard question 5); Evolutionary-Claude (concrete suggestion S6 "Pre-warm the derivation cache on workspace restore").

- **I6: Stale-worktree behavior unspecified — operator runs `git worktree remove` while a pane is open**
  - Where in the plan: Edge-case table covers "worktree pointing at a deleted *branch*" but not "worktree directory removed while a pane is open in it."
  - Problem: A common operator workflow is `git worktree remove <name>` from a different pane after merging, or `git worktree prune`. The affected pane's OSC 7 never fires again (no `cd` happened). The cache key (cwd, mtime-of-resolved-HEAD) — but the resolved HEAD file is now gone. `stat` returns ENOENT. Behavior is undefined; most likely the cache lookup fails, the resolver returns nothing or errors, and the chip either disappears or sticks with stale data forever.
  - Revision: Add to BUILDPLAN: "When the resolver finds that `cwd` no longer exists, or the resolved gitdir/HEAD path is missing, return a defined 'stale' result. The renderer either clears the chips (preferred) or renders them with a stale indicator (e.g., strikethrough/dimmed/red). Additionally, on every OSC 7 fire, the resolver checks whether the previously-cached cwd's filesystem state is still valid and re-runs if not." Add an AC for the test: "after creating a worktree, opening a fixture surface in it, running `git worktree remove`, and re-resolving, the resolver returns the defined stale result and the projection produces empty chips (no crash)."
  - Sources: Adversarial-Claude (blind spot #5 "deleted-worktree case" + reality stress test disruption 3 + hard question 4); Adversarial-Codex (blind spots "Bare repos and deleted worktrees are not defined rigorously" + early warning signs "A chip remains after `cd /tmp` or after deleting the worktree directory").

- **I7: Long-branch-name handling unspecified — branch ≤64 constraint conflicts with real-world git**
  - Where in the plan: SPEC table — `branch` "string, ≤64."
  - Problem: Git branch names can exceed 64 characters. The plan doesn't say whether the resolver should truncate at 64, reject (return nil) over 64, abbreviate (head…tail), or store full text and let the view layer truncate. A derived resolver should not fail to render simply because the real branch name is long.

Update AC9 (or add an AC) to assert a 200-char branch name renders without crash and with a truncated chip + full value queryable.
  - Sources: Standard-Codex (weakness #10 + question 9); Adversarial-Codex (hard question 14 implicit; flagged separately).
