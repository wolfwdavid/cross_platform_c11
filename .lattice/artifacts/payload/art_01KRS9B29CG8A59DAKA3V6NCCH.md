# Plan Review: C11-14 — Default terminal agent

## 1. Verdict

**FAIL (plan-level)** — The plan is well-shaped and the in-scope/deferred split is clean, but three concrete specification gaps will surface as bugs or rework if implementation starts now: (a) the test-target placement is ambiguous, (b) the initial-prompt delivery mechanism is wrong for interactive TUIs, and (c) the extraArgs / customCommand shell semantics are undefined. Each is a small revision; the overall direction is right.

## 2. Summary

Reviewed the plan for adding a configurable default terminal agent that launches on new terminal surfaces (with a "New Bash Terminal" escape hatch). The plan correctly identifies the file seams (`TerminalController`, `Workspace`, `TabManager`, settings page, menu wiring), names the precedence chain, defers genuinely-deferred work (named presets, lineage composition, per-workspace UI), and honors the "c11 is unopinionated about TUI config" principle. The critical concern is that the launch-mechanism details (how the prompt and args actually reach the agent process) are underspecified, and the test-target story conflates "directory" with "build-phase membership" in a way that's load-bearing for the "safe to run locally" promise.

## 3. Issues

**[CRITICAL] Tests (c11-logic only) — test-target placement is ambiguous and load-bearing**
The plan lists new test files under `c11Tests/DefaultAgentConfigTests.swift` and `c11Tests/DefaultAgentResolverTests.swift` while claiming "c11-logic only". Both targets in `GhosttyTabs.xcodeproj/project.pbxproj` (`c11LogicTests` at id `CD94D227…` and `c11Tests` at id `F1000004…`) pull from files in the `c11Tests/` directory — directory placement does NOT determine target membership; pbxproj build-phase membership does. If these files land only in the `c11Tests` (host) target's Sources phase, the "safe to run locally via `c11-logic`" promise is broken and they will end up needing the host-app launch that crashes the operator's running c11 (per CLAUDE.md's testing policy and the recent PR #164 incident memo). The xcodeproj-ruby-gem normalization gotcha (CLAUDE.md "Pitfalls") makes the diff-review path for catching this fragile.
**Recommendation:** State explicitly that the new test files are added to the `c11LogicTests` target's Sources build phase (PBXSourcesBuildPhase id `37DDE3B0A6A70E75A7B2BEDF`) and are NOT added to the `c11Tests` (host) target. Spell out the verification step: `xcodebuild -project GhosttyTabs.xcodeproj -scheme c11-logic test` must include the new tests; the host scheme should not list them under its source files. Consider also linking the files from a subdirectory like `c11Tests/Logic/` to make the intent visually obvious in the file tree.

**[CRITICAL] Resolution function — `initialPrompt` delivery via `<<<` is wrong for interactive TUIs**
The plan says non-empty `initialPrompt` is "piped via `<<<` if non-empty" and that codex/kimi/opencode take "initialPrompt as stdin if any". `<<<` is a bash herestring — it only works if the command runs through a shell, and for interactive TUIs (claude, codex) it pipes a closed stdin, which either drops the agent out of interactive mode or breaks the readline loop. Claude Code actually accepts an initial prompt as a trailing positional arg (`claude "prompt"`); codex has its own conventions; opencode/kimi differ again. There is no universal stdin-prompt convention here. This will fail the first time it's tried, and the "logic-only" tests can't catch it because it's an integration concern.
**Recommendation:** Replace the `<<<` mechanism with a per-agent strategy in the resolver:
- claude-code → trailing positional arg, quoted (`claude --model X "prompt"`)
- codex → check codex's actual interactive-prompt flag/positional convention before committing
- kimi / opencode → same — verify in a tagged build before locking the design
- custom → operator's responsibility (document it; the resolver passes the prompt through an env var like `C11_INITIAL_PROMPT` so the operator can interpolate where they want)

Run a manual smoke test in a tagged build for each agent type as part of the implementation pass and capture the working pattern in the plan before merge.

**[CRITICAL] Resolution function — shell quoting of `extraArgs` and `customCommand` is undefined**
The plan says `extraArgs` is "free-text additional flags, appended as-is" and `customCommand` is similarly unstructured. But "appended as-is" to what? An argv array passed to `posix_spawn`? A single shell-string handed to `/bin/sh -c`? Through the existing `remoteTerminalStartupCommand` channel (which appears to be a string)? An operator typing `--foo "bar baz"` will get different results in each model: argv array splits on whitespace and quotes are literal; shell-string is parsed by sh and quotes work; `posix_spawn` direct exec ignores quoting entirely. This needs pinning before code is written, both for correctness and for the security/footgun story (the operator owns the box, but the behavior must be documented for them to reason about).
**Recommendation:** Pick one model and document it. Recommended: build the launched command as a shell-string (since the existing `initialCommand: String?` on `TerminalPanel` looks string-shaped — verify), and document that `extraArgs` is parsed per shell-token rules (so quoted strings are honored). If you go argv-array, switch the data model to `extraArgs: [String]` rather than `String`. Either way, add a unit test that an extraArgs value of `--model "claude opus 4-7" --verbose` reaches the resolver correctly.

**[MAJOR] Workspace plumbing — `newTerminalSurfaceInFocusedPane` is missing from the change list**
The menu-driven New Terminal path (`TabManager.newSurface()` at `TabManager.swift:3568`) calls `Workspace.newTerminalSurfaceInFocusedPane(focus:)` at `Workspace.swift:9074`, which in turn calls `newTerminalSurface(inPane:focus:)`. The plan threads an `agentOverride` parameter through `newTerminalSurface` and `newTerminalSplit` but never mentions `newTerminalSurfaceInFocusedPane`. Without extending that signature too, the menu's "New Terminal" item cannot deliver the resolved agent override (it would need a separate call path or default to the un-overridden surface creation).
**Recommendation:** Add `Workspace.newTerminalSurfaceInFocusedPane` to the file-level change list. Either thread `agentOverride:` through it, or have `TabManager.newSurface()` resolve the agent once and call `newTerminalSurface(inPane:focus:agentOverride:)` directly with the focused pane id.

**[MAJOR] Workspace plumbing — relationship to existing `remoteTerminalStartupCommand` is undefined**
`Workspace.newTerminalSurface` at line 7860 already constructs `let remoteTerminalStartupCommand = remoteTerminalStartupCommand()` (line 7872), passes it as `initialCommand` to the new `TerminalPanel`, and conditionally calls `trackRemoteTerminalSurface(newPanel.id)` (line 7888). The same shape repeats in `newTerminalSplit` (lines 7755, 7792, 7796). The plan says "replace `remoteTerminalStartupCommand`" when an agent override is non-nil, but doesn't say what happens to the `trackRemoteTerminalSurface` tracking — should agent-launched surfaces be tracked the same way (and inherit whatever lifecycle behavior that tracking enables), or treated as a distinct category? Without this answer the override may silently regress remote-terminal behavior, or agent surfaces may end up untracked when they shouldn't be.
**Recommendation:** Add a short subsection clarifying: (a) does an agent override REPLACE `remoteTerminalStartupCommand` or COMPOSE with it (and what happens when both are present)?, (b) should `trackRemoteTerminalSurface` fire for agent surfaces, or is there an agent-tracking equivalent (e.g., `AgentRestartRegistry`)?, (c) what happens in a remote-workspace context where both signals exist?

**[MAJOR] Per-project config discovery — "cwd" is underspecified**
"Walk up from cwd looking for `.c11/agents.json`" — but for a new-surface call inside a running c11 app, "cwd" could mean: the c11 process's cwd, the focused surface's cwd, the workspace's working directory, or the operator's shell at the moment the socket command lands. These differ and produce different config files. The blueprint-store pattern referenced in the plan (`WorkspaceBlueprintStore.perRepoBlueprintURLs(cwd:)`) takes cwd as an explicit parameter precisely because the answer is non-obvious.
**Recommendation:** Pin down which cwd is used in each call path: (a) menu-driven New Terminal → probably the focused surface's cwd, falling back to workspace working dir, (b) CLI `--agent` flag via socket → the calling shell's cwd (the socket command should carry it; check whether the existing `new-surface` socket protocol already includes a cwd), (c) split from existing surface → the source surface's cwd. State this in the Resolution-function section.

**[MAJOR] Per-workspace override — JSON-in-a-metadata-string is awkward**
Storing a Codable `DefaultAgentConfig` blob in a single metadata value (`workspace.default_agent_inline = "<json>"`) fights the existing flat key/value metadata pattern. Setting it via CLI requires shell-escaping a JSON string; reading it back requires JSON parsing on every lookup; introspection via `c11 list-metadata` shows an unreadable blob. The existing `WorkspaceBlueprintStore` already demonstrates the pattern for "structured per-workspace config": a file (`.c11/blueprints/<name>.{json,md}`), not a metadata blob.
**Recommendation:** Consider two alternatives: (1) per-field metadata keys (`workspace.default_agent.type`, `workspace.default_agent.model`, `workspace.default_agent.extra_args`, …) so each field is independently inspectable and settable; or (2) a per-workspace file like `.c11/workspaces/<id>/agent.json` that uses the same `DefaultAgentConfig` codec as the user-level and project-level configs. Either is more durable than a JSON-in-a-string blob.

**[MINOR] Menu wiring — "a second menu item" wording undercounts existing items**
The plan says "add a second menu item" in the "New Surface" block at `c11App.swift:968`. That block (verified) is already a `Menu` containing three items: "New Terminal", "New Browser", "New Markdown". Adding "New Bash Terminal" makes it a fourth item, not a second.
**Recommendation:** Change wording to "add a fourth item alongside New Terminal / New Browser / New Markdown" so the implementer doesn't accidentally drop existing items. Also call out the keyboard-shortcut decision (should "New Bash Terminal" get a shortcut, or stay menu-only? If shortcut, what — `Shift+Cmd+T`?).

**[MINOR] `--agent=default` semantics are thin in this scope**
With named presets deferred, `--agent` accepts exactly one value (`default`), which means the flag's only added value over no-flag-at-all is "force an error if precedence resolves to bash" (per the plan). That's a real distinction, but the plan introduces the entire flag for it. Either commit to the small additional semantic, or drop `--agent` from this PR and ship only `--bash`. Otherwise reviewers will wonder why the flag exists with one value.
**Recommendation:** Either (a) drop `--agent` from the first PR and add it with named presets later, OR (b) state explicitly what error message appears when `--agent=default` resolves to bash, and add a logic test that exercises the error.

**[MINOR] AgentDetector and AgentRestartRegistry overlap is unmentioned**
`Sources/AgentDetector.swift`, `Sources/AgentChip.swift`, and `Sources/AgentRestartRegistry.swift` already exist for runtime agent detection, sidebar chips, and restart behavior. When the new default-agent resolver launches a claude/codex process, will those detectors fire correctly? Will the AgentChip show? Will the restart registry know how to restart this surface? The plan doesn't say, which means the implementer will discover the answer the hard way.
**Recommendation:** Add a sentence under "Workspace plumbing" or "Risk register" stating that AgentDetector picks up the launched process via its normal detection path (the plan says it does, since the wrapper at `Resources/bin/claude` already calls `c11 set-agent` in the background). If true, manual-verify in a tagged build during implementation. If not, the chip/restart story needs design work.

**[MINOR] `.cmux/agents.json` legacy fallback**
`WorkspaceBlueprintStore` walks for both `.c11/blueprints/` and legacy `.cmux/blueprints/`. The plan only mentions `.c11/agents.json` for the new feature, which is probably the right call (this is new, no legacy operators exist), but per the memory note "Keep upstream cmux features" the decision deserves an explicit line.
**Recommendation:** Add a one-liner under Per-project discovery stating that `.cmux/agents.json` is intentionally NOT checked because this feature is c11-only and has no upstream/legacy counterpart.

**[MINOR] CwdMode introduces a parallel concept**
`DefaultAgentConfig.cwdMode` (`.inherit` / `.fixed`) plus `fixedCwd: String` overlaps with the existing `workingDirectory: String?` parameter on `newTerminalSurface` (nil = inherit). The resolver should output a single `workingDirectory: String?` (nil = inherit, non-nil = fixed) and let the existing path handle nil-as-inherit; the cwdMode/fixedCwd two-field shape is only useful at the Settings-UI layer for the form, not at the resolution boundary.
**Recommendation:** Keep `cwdMode` + `fixedCwd` in the persisted `DefaultAgentConfig` struct if it makes the Settings UI cleaner, but state explicitly that `DefaultAgentResolver.resolve(...)` collapses them to a single `workingDirectory: String?` for downstream code.

## 4. Positive Observations

- **Clean in-scope/deferred split with operator-confirmed boundaries.** Listing named presets, lineage composition, and per-workspace UI as deferred (with an explicit "confirmed with operator 2026-05-16" note) prevents scope creep mid-implementation and gives reviewers a clear yardstick.
- **Honors the "c11 is unopinionated about TUI config" principle.** Explicit rejection of `c11 install <tui>` and clear statement that `Resources/bin/` wrappers are untouched. This is exactly the kind of constraint that's easy to drift on; calling it out in the plan locks it in.
- **Typing-latency hot paths explicitly cleared.** Risk register names `WindowTerminalHostView.hitTest`, `TabItemView`, `TerminalSurface.forceRefresh` and confirms no work touches them. This is the right kind of risk discipline for this codebase.
- **Localization workflow follows the established translator-sub-agent pattern** and correctly sequences it as the last step (after English strings stabilize). Matches CLAUDE.md guidance.
- **Acknowledges abandoned C11-14 branches.** The "Notes on the abandoned branches" section prevents anyone from accidentally resurrecting the CMUX-37 workspace-snapshot work that was orphaned in `ec4e69472`.
- **File-level change list with new/modify column.** Makes PR-time review tractable; reviewers know which files are new (deeper scrutiny) vs. modify (delta-only).
- **Implementation order is sound.** Data model + resolver + tests first (the part that's cheapest to iterate on via `c11-logic`), CLI flags next, UI and translator last. Matches the testing/iteration loop the codebase actually rewards.
- **All three `TerminalController` line references verified accurate** (`newSurface` at 18697, `newSplit` at 15358, `newPane` at 17158). The plan was clearly written against the current code, not from memory.
